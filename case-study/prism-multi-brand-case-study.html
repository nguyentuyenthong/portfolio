One library. Three brands.

A multi-brand color token architecture for Solaris Group, a fictional fintech with three product brands. Every neutral, text, and feedback color is locked across brands — only a small set of brand tokens swaps per theme. This page is the working demo: it re-themes itself.

01

The problem

Solaris Group runs three products for very different audiences: Solaris (everyday banking), Fern (green savings), and Ember (credit & rewards). The naive approach is one library per brand — same components, hard-coded colors. Every button change then has to be made three times, and the copies inevitably drift apart. The requirement I set: components are built once and never contain a hex value. A brand becomes nothing more than a set of values plugged into the same sockets.

02

Three tiers, one rule

The token system has three tiers. Only the last one has theme modes — that single decision is what makes the whole system safe.

TIER 1 · PRIMITIVES — NO MODES

Raw palettes

Context-free hex values: a neutral ramp plus one ramp per brand and per feedback role. Never applied to UI directly.

indigo/600 · green/600
orange/600 · gray/0–900
TIER 2 · SEMANTIC BASE — NO MODES

Shared foundation

Backgrounds, surfaces, borders, text, feedback. This collection has no modes, so no brand can ever bend it — consistency is structural, not disciplinary.

background · surface · border
text/primary · feedback/*
TIER 3 · SEMANTIC BRAND — 3 MODES

Theme tokens

Only the tokens that genuinely carry brand identity. Each mode re-aliases them to a different primitive ramp.

brand/primary · brand/tint
brand/accent · text/brand
primitives ──alias──▶ semantics ──apply──▶ components (built once)
03

What stays, what swaps

Before any theme exists, every token is sorted into exactly one of two buckets. If a token feels like it belongs to both, that's the signal to split it into two tokens.

Locked — semantic base
  • background · gray/0
  • surface · gray/50
  • border · gray/200
  • text/primary · gray/900
  • text/secondary · gray/500
  • feedback/error · red/500
  • feedback/success · green/400
  • feedback/warning · amber/500
Themed — semantic brand
  • brand/primary — the identity color
  • brand/primary-hover / -pressed
  • brand/on-primary — text on brand fills
  • brand/tint — soft brand surfaces
  • brand/accent — secondary highlight
  • text/brand — brand-colored text

Note that feedback colors live in the locked bucket. A success state is a system status, not a brand statement — if Fern's green were also the success green, error and brand would collide on Ember. Separating them is one of the quiet decisions that keeps the system honest.

05

The accessibility catch

The tempting shortcut when adding a brand is to grab the 500-level "hero" shade of its ramp — the one from the logo. But contrast has to be audited per mode. White text on Fern's and Ember's hero shades fails WCAG 2.1 AA for normal text (minimum 4.5:1), so both themes alias brand/primary one step darker. This is exactly why brand/primary is an independent token rather than "just use the logo color."

06

Playbook — onboarding a new brand

Add the primitive rampDraw the new brand's 50–800 ramp in the primitives collection. Nothing else is touched yet.

Audit contrast before aliasingFind the shade that clears 4.5:1 against white (or dark) text. That shade — not the logo shade — is the candidate for brand/primary.

Add a mode to the theme collectionDuplicate the default mode, then re-alias only the themed rows to the new ramp. In Figma this is one column; in CSS it's one attribute selector.

Leave the base collection aloneIt has no modes, so there is literally nothing to do. If a brand "needs" a different page background, that's a scope conversation — not a token edit.

Set the mode on the outermost frameEvery nested component instance inherits it and re-themes — no detaching, no manual overrides.

Re-run the WCAG checklist per modebrand/primary × on-primary, tint × text, focus ring × background. Only then publish.
