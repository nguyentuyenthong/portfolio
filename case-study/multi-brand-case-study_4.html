One library. Three brands. Light & dark. Zero duplicated files.

A two-dimensional token architecture for Solaris Group, a fictional fintech with three product brands. Brand picks the color ramp; theme picks the stop. Neutrals, text, and feedback are locked across brands — and this page is the working demo: switch either dimension below and the entire case study re-themes itself.

01

The problem

Solaris Group runs three products for very different audiences: Solaris (everyday banking), Fern (green savings), and Ember (credit & rewards) — and every product ships both a light and a dark theme. The naive approach is one library per brand per theme: six copies of every component, drifting apart with every edit. The requirement I set: components are built once and never contain a hex value. A brand is a set of values plugged into the same sockets — and a theme is a second, independent set of sockets. The two must never be multiplied together.

02

Two dimensions, three collections

The trap with brand + theme is building one collection with six modes (Solaris-Light, Solaris-Dark, Fern-Light…). Every new brand then costs two columns, every new theme costs three, and someone will eventually "fix" a neutral in one cell. Instead, the dimensions stay orthogonal — each in its own collection, chained by aliases.

COLLECTION 1 · PRIMITIVES — NO MODES

Raw palettes

Context-free hex ramps: neutrals plus one ramp per brand and per feedback role. Never applied to UI directly.

COLLECTION 2 · BRAND RAMP — 3 MODES

Brand picks the ramp

Defines only an abstract ramp — brand/ramp/50…950 — re-aliased per brand mode. No background, no text, no button color lives here.

Fern → green/* · Ember → orange/*
COLLECTION 3 · SEMANTICS — 2 THEME MODES

Theme picks the stop

Every UI-facing token lives here, base and brand alike. Light aliases brand/primary → ramp/600; Dark → ramp/400. Base flips gray/0 ↔ gray/925. Components consume only this collection.

background · surface · text/*
brand/primary · brand/tint · on-primary
primitives ──▶ brand ramp (brand mode) ──▶ semantics (theme mode) ──▶ components (built once)
The mode math: 3 brands × 2 themes = 6 visual expressions, served by only 3 + 2 = 5 mode columns. In the flattened single-collection approach it would be 6 columns today — and 8 the moment a fourth brand arrives. In Figma, a frame simply carries two independent mode settings: one for the Brand ramp collection, one for the Semantics collection.
03

What moves on which axis

Every token is classified against both dimensions before any theme exists. The matrix is the contract: if a token seems to need a cell it wasn't given, that's a design conversation — not a token edit.

text/primary · border—✓ flips with themeReadability is theme physics, not identity

Notice the asymmetry: no token changes with brand only once dark mode exists — anything brand-colored must also answer to the theme, because contrast physics differ on dark canvases. This is exactly why brand values are expressed as an abstract ramp instead of a single "brand color".

05

The accessibility catch — on both axes

Contrast has to be audited per combination. On the brand axis, the tempting shortcut is the hero shade from the logo — white text on Fern's and Ember's hero shades fails WCAG 2.1 AA for normal text (≥ 4.5:1), so light themes alias one step darker. On the theme axis, dark mode lightens the brand fill for legibility on dark canvases — and then white text fails on the brand's own color. That's why brand/on-primary is themed instead of hard-coded white.

06

Playbook — extending either axis

Add the primitive rampA new brand starts as a 50–950 ramp in the primitives collection. Nothing else is touched yet.

Audit contrast at both endsFind the stop that clears 4.5:1 under white text (light-theme candidate) and the stop that carries dark text on a dark canvas (dark-theme candidate). Usually ~600 and ~400.

Add one mode to the Brand ramp collectionRe-alias brand/ramp/50…950 to the new ramp. One column. The Semantics collection doesn't change at all — it already knows how to read a ramp.

Never multiply the axesA new theme (e.g. high-contrast) is one mode in Semantics; a new brand is one mode in Brand ramp. If you ever find yourself creating a "Fern-Dark" mode, the architecture has been flattened — stop and refactor.

Leave the theme-axis base alone when brandingBackgrounds, text and borders have no brand modes, so there is literally nothing to edit. If a brand "needs" its own canvas color, that's a scope conversation, not a token edit.

Set both modes on the outermost frameIn Figma the frame carries two independent mode settings — brand and theme. Every nested instance inherits both and re-themes with no detaching.

Re-run WCAG per combinationbrand/primary × on-primary, tint × text, focus ring × background — for each brand in each theme. Six checks, automated where possible, before publishing.
