Fastbuy is a device trade-in ecosystem — running the entire lifecycle from appraising used devices, pricing, and creating orders, to managing partners and the Samsung Trade In Plus program. It's a large B2B product with many roles and many interconnected business modules, so the core challenge isn't one beautiful screen — it's keeping the whole system consistent and easy to scale as the number of screens grows fast.

I joined as the UI designer and component-system owner — making sure ~10 modules, despite different business logic, still speak the same “visual language”: the same data tables, the same metric cards, the same status badges, the same detail-screen layout.

02

Context & users

Fastbuy isn't a single screen — it's an operational system that runs every day in stores and in the reconciliation team. As the product swelled to ~10 modules with hundreds of screens, the biggest challenge wasn't “is this screen pretty” but “does the whole system still speak the same language”. That's where a good design system makes the difference.

The journey of one trade-in order

An order passes through several modules in sequence. Design's job is to make these stages flow smoothly into one another — same patterns, same way of reading status — even though each stage is a distinct business function.

  1. Login & permissions

    Staff enter the system through the auth flow (login, OTP, forgot password) and see the right modules for their role.

  2. Appraise the used device

    Enter the IMEI, run the condition checklist, attach photos/video — the multi-layered detail screen is the core of the whole system.

  3. Price against the price list

    The system looks up the price list by device model & condition to produce the buy-back price, with the ability to edit and duplicate price lists.

  4. Create & process the order

    The trade-in order is created and runs through its statuses — operations track progress on the same dashboard pattern.

  5. Reconciliation & partners

    The order is reconciled with partners/stores and the Samsung Trade In Plus program.

  6. Promotions & vouchers

    Vouchers attach to orders with a validity window, used/remaining counts and a clear status.

04

Design system

Design the system, not each screen

For a large product, polishing each screen isn't enough. I started from the repeating blocks — data tables, metric cards, status badges, detail-screen layout, empty states — then standardized them into components to assemble across every module.

Consistency is a feature

When 10 modules share one visual language, users learn it once and use it everywhere, and the dev team ships new screens far faster. Color/spacing tokens keep everything aligned as the system grows.

One dashboard pattern for every module

The same frame — opening metric cards, filters, then a data table with status badges — is reused from appraisal to orders, vouchers and Samsung Trade In Plus. Only the columns and metrics change per business function; everything else stays the same.

Key design decisions

Decision Rationale Trade-off
One shared data-table template Appraisal, orders, vouchers, pricing… are all tabular data — one pattern makes scanning fast and maintenance easy Needs flexible column configuration to fit each business function
Dashboards open with metric cards Operators need to see the situation immediately (total orders, successful, cancelled) before diving into the list Takes up space at the top of the screen — acceptable given the fast-scan value
Detail layout organized by information layers The appraisal screen is very data-heavy — split into product info → classification → checklist → photos/video so it reads in one flow The screen is long, so it needs clear anchors/headings to avoid getting lost
Semantic status color system Order/voucher/appraisal statuses map to fixed colors so the situation reads instantly Must limit the number of statuses so the palette doesn't get diluted

A shared status system

Orders, appraisals, vouchers and price lists all have a “situation” that needs to be read quickly. Instead of each module inventing its own colors, I mapped everything to one set of semantic badges — users understand by color, without reading the text.

Badge Meaning Example usage
Success Completed, valid, active Completed order · voucher still valid
Processing Awaiting action, in progress Order awaiting appraisal · awaiting reconciliation
New Just created, not yet processed New appraisal ticket · newly created order
Draft / Paused Not applied, paused, expired Draft price list · expired voucher
Rejected / Cancelled Rejected, cancelled, invalid Cancelled order · rejected ticket

AI in the workflow

With such a large volume of screens, I used AI to speed up the repetitive parts: auditing frame/layer naming and syncing components across modules, quickly drafting microcopy for statuses and empty states, and cross-checking consistency between tables. That let me spend time on what really needs design judgment — information structure and the overall experience.

UI & principles

A few representative screens

The screens below belong to very different business functions, yet they share the same layout, tables, buttons and badges — that's exactly what a design system brings to a large product.

Principles I took away

① System first, screens second

In a large product, the value is in standardized repeating blocks — not one nice one-off screen. Building the patterns first makes every later screen fast and aligned.

② Consistency is a feature

When ~10 modules speak one visual language, users learn it once and use it everywhere, and the dev team ships new screens far faster.

③ Dense data needs structure, not decoration

The appraisal detail screen is clarified with information layering and whitespace, rather than adding color or dividers — faster to read while staying tidy.

④ Tokens keep the system aligned as it grows

Semantic colors, spacing and components all come from tokens. That way, each new module snaps right into the rest of the product.

If I did it again

  • Build the component library earlier Standardizing tables and detail screens from the start would save many rebuild cycles as modules grow fast.
  • Centralize status documentation A shared table of status color & label conventions keeps new modules from “reinventing” them.

Next steps
